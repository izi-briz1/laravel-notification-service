## ⚙️ Функциональные требования (Бизнес-требования)

- **Массовая рассылка уведомлений.** Система должна предоставлять API для запуска массовой отправки SMS или Email-сообщений. Инициатор запроса передаёт канал связи, текст сообщения и массив идентификаторов получателей.
- **Приоритизация трафика.** Система должна гарантировать доставку критичных сообщений без задержек. Транзакционные уведомления (коды доступа, срочные изменения маршрутов) должны получать наивысший приоритет и отправляться вне очереди, обгоняя маркетинговые рассылки.
- **Детализация статусов доставки.** Система должна предоставлять API для запроса истории и текущего статуса всех уведомлений конкретного подписчика. Статусы:
    - **в очереди** - сообщение принято и ожидает отправки;
    - **отправлено** - передано шлюзу/провайдеру;
    - **доставлено** - подтверждено провайдером;
    - **отброшено** - ошибка доставки, несуществующий номер/email и т. д.

## Нефункциональные требования (Архитектура и качество)

- **Гарантия доставки сообщений (Reliability):**
    - *Персистентность:* использование брокеров сообщений для хранения очереди.
    - *Модель доставки:* поддержка семантики **at-least-once**. Огромным плюсом будет реализация **exactly-once** на уровне бизнес-логики.
    - *Retry-механизмы:* автоматический повтор попыток отправки при временной недоступности шлюзов.
- **Дедупликация (Idempotency):** защита от повторной отправки одного и того же сообщения при дублировании запросов от вызывающего сервиса.
- **Тестирование:** обязательное наличие интеграционных тестов на основные сценарии. Мы хотим видеть автоматизированную проверку всей цепочки: от получения сообщения из очереди до корректного изменения статуса в базе данных и вызова нужного провайдера.
- **Cloud Native и развертывание:**
    - Упаковка сервиса в Docker-образ.
    - Весь проект (БД, брокер, кэш, приложение) должен запускаться одной командой `docker-compose up`.

# Notification Service

Микросервис массовой рассылки SMS и Email. Принимает рассылку через REST API, раскладывает по очередям RabbitMQ (транзакционные сообщения едут вне очереди, обгоняя маркетинговые), отправляет через провайдера и ведёт полную историю статусов каждого уведомления. Главный фокус - надёжность: сообщение не должно потеряться, даже если в неудачный момент упадёт воркер, брокер или само приложение.

## Быстрый старт

Нужен только Docker (Compose v2+).

**Стек:** PHP 8.4 / Laravel 12, PostgreSQL 17, RabbitMQ 4, Redis 7, Docker Compose.

```bash
git clone <repo-url>
cd laravel-notification-service
docker-compose up -d --build
```

Это всё - одна команда поднимает весь стек, миграции применяются сами при старте.

| Сервис | Адрес |
| --- | --- |
| API | http://localhost:8080/api/v1 |
| Swagger UI | http://localhost:8080/api/documentation |
| RabbitMQ Management | http://localhost:15672 (guest/guest) |

Тесты гоняются против настоящих PostgreSQL и Redis, никаких моков базы:

```bash
docker-compose run --rm -e RUN_MIGRATIONS=false app php artisan test
```

## Два краеугольных камня

В системе есть два места, где процесс может упасть «посередине» - и именно вокруг них построена вся надёжность. Чтобы разобраться в механике, достаточно посмотреть три файла.

**1. Упали между записью в базу и отправкой в RabbitMQ.** Уведомление уже сохранено в PostgreSQL со статусом `queued`, но до публикации в брокер процесс не дожил (или сам RabbitMQ был недоступен). Воркеры про такое сообщение просто не узнают. Поэтому при успешной публикации запись помечается `dispatched_at`, а scheduler раз в минуту находит записи без этой пометки и отправляет их в брокер повторно. Если получится дубль публикации - не страшно, воркер его отбросит (см. пункт №2).

**2. Зависли в статусе `sending`.** Воркер взял уведомление в работу (`queued -> sending`) и умер жёстко - kill -9, OOM. Статус так и остался `sending` навсегда. Брокер вернёт сообщение в очередь, но воркер его отбросит как дубль - тот самый exactly-once гард, который защищает от повторной отправки. Поэтому отдельная команда раз в минуту ищет уведомления, висящие в `sending` слишком долго, и возвращает их в очередь. Да, в этом редком случае человек может получить сообщение дважды. Но так как у нас есть transactional-сообщения - коды доступа, входы - пусть лучше человек получит их 2 раза, чем не получит вообще.

**Куда смотреть:**

| Файл | Что внутри |
| --- | --- |
| [`app/Jobs/SendNotificationJob.php`](app/Jobs/SendNotificationJob.php) | Воркер: захват `queued -> sending` (защита от дублей), вызов шлюза, ретраи с нарастающей задержкой |
| [`app/Console/Commands/DispatchStuckNotifications.php`](app/Console/Commands/DispatchStuckNotifications.php) | Камень №1: дослать в брокер то, что сохранилось в базе, но не опубликовалось |
| [`app/Console/Commands/RequeueStuckSendingNotifications.php`](app/Console/Commands/RequeueStuckSendingNotifications.php) | Камень №2: вернуть в очередь то, что зависло в `sending` после смерти воркера |

## Архитектура

Путь сообщения от запроса до доставки:

```
Клиент --POST /api/v1/notifications (Idempotency-Key)--> API (nginx + php-fpm)
   │            дедубликация: Redis SETNX + UNIQUE(idempotency_key) в PG
   │            batch + N notifications (status=queued) - одна транзакция PG
   │            после коммита: публикация jobs в RabbitMQ + пометка dispatched_at
   │            (scheduler раз в минуту добирает неопубликованные - outbox-свипер)
   │
   │   RabbitMQ: очередь `transactional`   очередь `marketing`
   │                  |                          │
   │     worker-transactional          worker-marketing
   │                  |----------|---------------|
   │                    SendNotificationJob:
   │                    1. атомарный захват queued->sending (exactly-once гард)
   │                    2. rate limit канала (Redis::throttle, общий на воркеры)
   │                    3. вызов шлюза (mock SMS/Email провайдера)
   │                    4. sent | ретрай с backoff | failed
   │
   │   Провайдер --DLR--> POST /api/v1/webhooks/delivery --> delivered | failed
   │
   └--GET /api/v1/subscribers/{id}/notifications--> история + статусы
```

### Из чего состоит docker-compose

| Сервис | Роль |
| --- | --- |
| `app` (php-fpm) + `nginx` | HTTP API |
| `worker-transactional` | Воркер, который слушает только очередь `transactional` |
| `worker-marketing` | Воркер очереди `marketing` |
| `scheduler` | Laravel Scheduler - каждую минуту запускает свиперы из «краеугольных камней» |
| `postgres`, `rabbitmq`, `redis` | База, брокер, кэш/лимиты |

### Жизненный цикл уведомления

`queued` (приняли, ждёт отправки) -> `sending` (внутренний статус: воркер прямо сейчас вызывает провайдера) -> `sent` (передали шлюзу) -> `delivered` (провайдер подтвердил) или `failed` (не доставили: ошибка, несуществующий адресат, кончились попытки).

Каждый переход записывается в `notification_status_history` - когда, откуда куда и почему. Вся хронология отдаётся через API.

## Гарантии доставки

**At-least-once из коробки.** Очереди RabbitMQ durable, сообщения persistent, ack уходит только после обработки. Где бы ни упал воркер, сообщение вернётся в очередь, а не пропадёт.

**Никаких потерь между базой и брокером.** Рассылка и все её уведомления сохраняются одной транзакцией, публикация в RabbitMQ идёт после коммита. Если между этими шагами что-то упало - записи без пометки `dispatched_at` доберёт свипер (камень №1). Выборка дешёвая: частичный индекс по `status=queued AND dispatched_at IS NULL`.

**Exactly-once на уровне бизнес-логики.** Брокер может доставить сообщение дважды - это нормально для at-least-once. Но провайдер из-за этого дважды вызван не будет: воркер сначала захватывает уведомление условным `UPDATE ... SET status='sending' WHERE id=? AND status='queued'`, и при повторной доставке этот UPDATE вернёт 0 строк - джоба просто молча выйдет. Все смены статусов идут через одну точку (`NotificationStatusService`), которая следит за допустимыми переходами.

**Если воркер умер прямо во время отправки** - это камень №2, его лечит второй свипер: вернёт зависшие в `sending` обратно в очередь, а исчерпавшие попытки переведёт в `failed`. Здесь возможен дубль у получателя, и это осознанный выбор: пусть лучше код доступа придёт два раза, чем не придёт вообще.

**Идемпотентность API.** Заголовок `Idempotency-Key` обязателен. Быстрая проверка - Redis `SET NX` (TTL 24 часа), но Redis может потерять ключ, поэтому источник истины - UNIQUE-констрейнт в PostgreSQL. Прислали тот же запрос второй раз - получите `200` с той же рассылкой вместо `201`, ничего не задублируется.

**Ретраи.** Если шлюз ответил временной ошибкой (`TransientGatewayException`) - возвращаем в очередь с нарастающей задержкой: 5с, 10с, 20с и так далее, максимум 5 попыток (счётчик в БД, колонка `attempts`). Если ошибка неустранимая (`PermanentGatewayException` - номер не существует, email невалидный) - сразу `failed`, ретраить бессмысленно.

**Лимиты (Redis).**
- *Исходящий*: шлюзы не любят, когда в них льют без ограничений, поэтому есть общий на все воркеры лимит по каналу (`Redis::throttle`, настраивается через `SMS_RATE_LIMIT_PER_SECOND` / `EMAIL_RATE_LIMIT_PER_SECOND`). Упёрлись в лимит - джоба вернётся в очередь, не сжигая бизнес-попытку.
- *Входящий*: rate limiter на API (`API_RATE_LIMIT_PER_MINUTE`, по умолчанию 120 запросов в минуту с IP).

**Приоритизация.** Очереди `transactional` и `marketing` физически разные, и у транзакционной - свои выделенные воркеры. Даже если в маркетинговой очереди лежит миллион сообщений «Скидки!», код доступа уйдёт немедленно. Воркеры масштабируются независимо: `docker-compose up -d --scale worker-marketing=4`.

## API

Полное описание - в Swagger UI: http://localhost:8080/api/documentation (генерируется автоматически при старте). Ниже - основные сценарии.

### Запустить рассылку

```bash
curl -X POST http://localhost:8080/api/v1/notifications \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -H "Idempotency-Key: demo-key-1" \
  -d '{
    "channel": "sms",
    "priority": "transactional",
    "text": "Ваш код доступа: 1234",
    "recipient_ids": ["user-1", "user-2", "user-3"]
  }'
```

`201` - принято. Повторите с тем же `Idempotency-Key` - получите `200` и ту же самую рассылку. Идентификаторы получателей принимаются как есть: за валидность номеров и адресов отвечает вызывающий сервис.

### Узнать статусы уведомлений подписчика

```bash
curl "http://localhost:8080/api/v1/subscribers/user-1/notifications?status=delivered&channel=sms&per_page=25" \
  -H "Accept: application/json"
```

Вернётся пагинированный список: текущий статус каждого уведомления плюс полная история переходов (`status_history`).

### Сводка по рассылке

```bash
curl http://localhost:8080/api/v1/notifications/batches/{batch_id} -H "Accept: application/json"
# -> {"data": {...}, "status_counts": {"delivered": 3}}
```

### DLR-webhook от провайдера

```bash
curl -X POST http://localhost:8080/api/v1/webhooks/delivery \
  -H "Content-Type: application/json" \
  -d '{"provider_message_id": "<uuid из sent>", "status": "delivered"}'
```

`200` - статус обновлён, `404` - такой id не знаем, `409` - уведомление уже в финальном статусе, менять поздно.

## Провайдеры-заглушки

Настоящих шлюзов нет - их имитируют `FakeSmsGateway` и `FakeEmailGateway`:

- генерируют `provider_message_id` и пишут «отправку» в лог;
- через несколько секунд сами подтверждают доставку, как это сделал бы DLR-колбэк настоящего провайдера, - так что статусы доезжают до `delivered` без ручных действий;
- умеют ломаться по заказу - удобно смотреть на ретраи вживую (настройки в `docker-compose.yml`):

| Переменная | Что делает |
| --- | --- |
| `FAKE_PROVIDER_TRANSIENT_FAILURE_PERCENT` | % временных ошибок шлюза - чтобы посмотреть на ретраи |
| `FAKE_PROVIDER_PERMANENT_FAILURE_PERCENT` | % неустранимых ошибок - несуществующий адресат |
| `FAKE_PROVIDER_AUTO_CONFIRM` | автоподтверждение доставки (`true` по умолчанию) |
